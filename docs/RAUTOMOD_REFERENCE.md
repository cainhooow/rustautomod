# `.rautomod` Reference

This document describes how Rust AutoMod resolves configuration and what each key does.

## Resolution order

Rust AutoMod resolves configuration like this:

1. Start from the current Rust file or generated target.
2. Walk upward looking for `.rautomod` files.
3. Parse the nearest matching config and its `extends=...` chain.
4. Evaluate rules from top to bottom.
5. Use the first rule whose `pattern` matches and is not excluded by negative patterns.
6. Mark the file as ignored if `exclude=` matches.
7. Fall back to VS Code settings under `rustautomod.*` if no `.rautomod` rule matches.

## File structure

A `.rautomod` file is split into rule blocks separated by blank lines.

Example:

```rautomod
schema_version=1
strict=warn
visibility=pub
sort=alpha

pattern=internal,!tests
visibility=private
target=mod.rs
```

The first block above acts like the default rule. The second block overrides behavior for matching files.

## Global keys

These keys are usually most useful in the first block.

### `schema_version`

Supported values:

- `1`

Purpose:

- pins the config format
- makes future migrations safer

Example:

```rautomod
schema_version=1
```

### `strict`

Supported values:

- `off`
- `warn`
- `error`

Purpose:

- controls how diagnostics are treated
- with `error`, invalid config diagnostics can block automod actions that rely on the config

Example:

```rautomod
strict=warn
```

### `extends`

Supported values:

- relative or absolute paths to another `.rautomod`-style file

Purpose:

- reuse shared rules
- keep workspace defaults in one place

Example:

```rautomod
extends=./shared.rautomod
```

Notes:

- relative paths resolve from the current `.rautomod`
- circular references are diagnosed as errors

## Rule keys

### `pattern`

Purpose:

- limits a rule to specific files or folders

Examples:

```rautomod
pattern=internal
pattern=src/api/**
pattern=core,!tests
```

Notes:

- comma-separated
- supports negation with `!`
- if omitted, the rule acts like a default rule

### `exclude`

Purpose:

- marks matching files or folders as ignored by Rust AutoMod

Examples:

```rautomod
exclude=generated/**
exclude=fixtures/**,snapshots/**
```

Notes:

- the config can still resolve successfully
- the file is simply treated as ignored for writes

### `visibility`

Supported values:

- `pub`
- `private`
- `pub(crate)`
- `pub(super)`

Purpose:

- chooses how generated `mod` declarations are written

Examples:

```rautomod
visibility=pub
visibility=private
visibility=pub(crate)
```

### `sort`

Supported values:

- `alpha`
- `alpha_case_insensitive`
- `none`
- `pub_first`
- `cfg_first`

Purpose:

- controls declaration ordering inside the managed block

Behavior summary:

- `alpha`: alphabetical by module name
- `alpha_case_insensitive`: alphabetical ignoring case first
- `none`: preserve insertion order
- `pub_first`: public module declarations before private ones
- `cfg_first`: cfg-managed declarations before plain ones

### `fmt`

Supported values:

- `enabled`
- `disabled`

Purpose:

- runs `cargo fmt` after a write when enabled

### `target`

Supported values:

- `auto`
- `mod.rs`
- `lib.rs`
- `main.rs`

Purpose:

- chooses which file receives the generated declaration block

Guidance:

- `auto` is usually the best default
- `mod.rs` is useful for folder-local module trees
- `lib.rs` is useful for public library crates
- `main.rs` is useful for binary crates

### `cfg`

Purpose:

- attaches one or more `#[cfg(...)]` lines to generated declarations

Examples:

```rautomod
cfg=unix,windows
cfg=feature="serde",all(unix, target_pointer_width = "64")
```

### `group_order`

Supported values:

- `use`
- `cfg`
- `pub_mod`
- `mod`
- `pub_use`

Purpose:

- controls ordering between managed declaration groups

Example:

```rautomod
group_order=use,cfg,pub_mod,mod,pub_use
```

Meaning:

- `use`: regular use statements
- `cfg`: module declarations with cfg attributes
- `pub_mod`: public module declarations
- `mod`: private module declarations
- `pub_use`: generated re-exports

### `blank_lines`

Purpose:

- controls how many blank lines are inserted between managed groups

Example:

```rautomod
blank_lines=1
```

### `reexport`

Supported values:

- `enabled`
- `disabled`

Purpose:

- adds `pub use self::<module>::*;` lines for matching declarations

Useful for:

- preludes
- facade modules
- grouped public APIs

### `header`

Purpose:

- adds a header comment at the top of the file content managed by the editor logic

Example:

```rautomod
header=generated by rustautomod
```

### `generated_comment`

Purpose:

- adds a comment line immediately before the managed declaration block

Example:

```rautomod
generated_comment=managed by rustautomod
```

Useful when:

- you want to make generated sections obvious in PRs
- multiple people edit the same module files manually

## Editor support

The `.rautomod` language integration currently provides:

- syntax highlighting
- dedicated file icon
- document formatting
- completions for supported keys and values
- diagnostics
- quick fixes for invalid keys and common missing keys

## Command-related settings

### `rustautomod.previewBeforeApply`

Supported values:

- `true`
- `false`

Purpose:

- when `true`, Rust AutoMod previews and asks for confirmation before writing changes

Example:

```json
{
  "rustautomod.previewBeforeApply": true
}
```

## Example configurations

### Basic library

```rautomod
schema_version=1
strict=warn
visibility=pub
sort=alpha
fmt=enabled
target=auto
```

### Binary application

```rautomod
schema_version=1
strict=warn
visibility=private
sort=alpha
target=main.rs
```

### Internal folders and generated code

```rautomod
schema_version=1
strict=warn
visibility=pub
sort=alpha
exclude=generated/**,fixtures/**

pattern=internal,!tests
visibility=private
target=mod.rs
sort=alpha_case_insensitive
```

### Facade module

```rautomod
schema_version=1
strict=warn
pattern=prelude
visibility=pub
reexport=enabled
group_order=cfg,pub_mod,mod,pub_use,use
generated_comment=managed by rustautomod
```

For end-to-end scenarios, see [USE_CASES.md](USE_CASES.md).
